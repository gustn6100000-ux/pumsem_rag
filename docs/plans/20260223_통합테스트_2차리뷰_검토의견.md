## 확인 사항 3건 — 이견 없음

**이슈 1 (TC10 미발화)**
점수 산출 정확. `톤`이 regex 목록에 없는 것까지 잡아낸 것이 핵심.

**이슈 3 (Phase 4 플랫 테이블)**
TC5 실제 응답 텍스트로 확인된 것이 가장 중요. `| 플랜트 용접공 | 0.287 | 인 | 200, SCH 40 |` 형식으로 Phase 4 적용 실증됨.

**이슈 4 (TC5 20.6s)**
현 시점에서 특정 불가라는 판단 타당. `search_info`에 단계별 타이밍 추가 방향도 맞음.

## 검토 의견 2건

### 의견 1 — TC9 Route 3.5 발화 근거 표현 불일치

문서에 "TC9: 4pt → complex 확정"이라고 되어 있으나, 이는 **코드 논리상 확정이지 실행 확인이 아님.**
이슈 2 결론표에서 서버 로그 항목을 "❌ 미확인"으로 정직하게 표기한 것과 일관성이 맞지 않음.

```
현재:  "TC9: 4pt → complex 확정"
수정:  "TC9: 코드 분석상 4pt → complex (서버 로그 미확인)"
```

---

### 의견 2 — `classifyComplexity` 개선 제안 중 `산출` +2pt 가중치 재검토

`합산|포함해서|전체|총액|계산해`에 `산출`을 추가하면 +2pt가 부여됨.
그러나 "노무비 산출"은 단순 규격 검색에서도 흔히 쓰이는 표현이어서 과분류 위험이 있음.

**과분류 시나리오:**
```
질의: "강관용접 200mm SCH 40 노무비 산출"  (단순 규격 검색)
현재:  1(길이) + 0 + 0 + 1(200mm) = 2pt → simple ✅ 올바름
산출 추가 후: 1 + 0 + 0 + 1 + 2(산출) = 4pt → complex ⚠️ 과분류
```

`산출` 단독보다는 다중 공종 컨텍스트에서만 complex로 보는 것이 더 정확함.

**개선 항목별 권고:**

| 개선 항목 | 원안 | 권고안 |
|---|---|---|
| `톤` regex 추가 | `톤` → +1pt | ✅ 그대로 적용 |
| `+` 연산자 | `+` 포함 → +1pt | ✅ 그대로 적용 |
| `산출` 추가 | `산출` → +2pt | ⚠️ +1pt로 낮추거나 "산출 + 공종 2개 이상" 조건부로 변경 권고 |
