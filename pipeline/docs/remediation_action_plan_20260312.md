# 품셈 데이터 검증 시스템: 향후 권장사항 세부 실행 계획서

> **개요**: 이전 교차 검증 및 복구 작업(`execution_report.md` 참조)을 통해 도출된 핵심 권장사항 4가지를 실제 개발 환경에 적용하기 위한 구체적인 액션 플랜(Action Plan)입니다.

---

## 1. `dedup_tables.py` 정밀화 재작성 (중복 오삭제 방지)

### 1-1. 목표 
헤더와 행 단위의 추상적 비교가 아닌, **테이블 내 실제 데이터 값의 100% 동일성**을 확인하는 방향으로 로직을 고도화하여 정상적인 테이블(페이지 분할, 연속 데이터)이 삭제되는 치명적 버그 수정.

### 1-2. 세부 실행 계획
*   **파일 수정**: `pipeline/scripts/dedup_tables.py`
*   **변경 로직**: 
    1.  `table_signature(tbl)` 함수의 시그니처 생성 방식 변경.
        *   기존: 상위 4개 헤더의 문자열 연결 + 행 수 비교
        *   개선: 테이블의 `headers` 배열 전체와 `rows` 배열 안의 모든 데이터를 `json.dumps(..., sort_keys=True)` 처리한 후, 해당 문자열의 SHA-256 해시를 고유 식별자(Signature)로 사용.
    2.  같은 `base_section_id` 내에 완벽하게 일치하는 해시를 가진 테이블이 두 개 이상일 경우에만 실제 중복으로 간주.
    3.  삭제 전 `DRY RUN` 모드를 기본 옵션으로 강화하여, 어떤 테이블이 중복 삭제되는지 터미널에 명시(표의 1행 데이터 추출 표기 등) 후 수동 승인.

## 2. `chunks.json` 자동 백업 파이프라인 구축 (Safety Net 확보)

### 2-1. 목표
`chunks.json` 파일이 DB 복구에 결정적인 백업 역할을 했으나, 향후 파이프라인(Phase 1) 재실행 시 무심코 덮어써져 손실되는 위험을 원천 차단.

### 2-2. 세부 실행 계획
*   **파일 수정**: `pipeline/phase1_preprocessing/step4_chunker.py` 또는 파이프라인을 실행하는 `run_pipeline.py`.
*   **로직 추가**:
    1.  `chunks.json`을 저장하는 로직 직전에 파일 롤오버(Rollover) 시스템 구현.
    2.  `phase1_output/archive/` 폴더 자동 생성.
    3.  저장 시 `chunks_YYYYMMDD_HHMMSS.json` 포맷으로 백업 복사본을 항상 보관하도록 모듈화.
    4.  아카이브 폴더 용량 관리 로직 추가 (최근 10개의 버전만 유지하고 이전 버전은 삭제하는 Cleanup 스크립트 작성).

## 3. 잔여 100건 누락 테이블 수동 파싱 복구 (Edge Case 대응)

### 3-1. 목표
자동화 스크립트(`fix_missing_tables.py`)로도 복구하지 못한 상위 누락 섹션(`5-3-7`, `5-4-2`, `5-4-4`, `10-3-1` 등 총 100개 테이블)을 수동 스크립트 대응 및 파서 예외 처리를 통해 완전 복구(Coverage 향상을 목표).

### 3-2. 세부 실행 계획
*   **1단계: 태그 구조 파악 (Root Cause)**
    *   에디터를 통해 저조한 수치의 원본 마크다운(예: `download_file` 내 섹션 `5-3-7` 영역)을 엽니다.
    *   HTML `<table>` 태그 내에 열림/닫힘 불일치, 복잡한 중첩된 `<td>` 또는 과도한 `colspan/rowspan`으로 파서가 강제 종료되거나 빈 배열을 리턴한 원인을 찾습니다.
*   **2단계: 특수 파서/보완 스크립트 작성**
    *   `fix_edge_case_tables.py`와 같이 특정 섹션만을 타겟팅하는 파이썬 스크립트 개발.
    *   원시 텍스트 정규식 치환(예: `</tr>` 누락 부분 강제 주입, 또는 쓰레기 문구 제거) 후 `BeautifulSoup`에 파싱을 넘기는 하이브리드 파싱 방식 채택.
    *   파싱된 객체를 직접 DB의 `graph_chunks` 해당 `chunk_id`에 업데이트.

## 4. 정기 감사 (Automation Testing) 구축

### 4-1. 목표
향후 품셈 데이터가 업데이트 되더라도 데이터 분실 퇴행(Regression)이 없도록, DB 적재와 마지막 단계 사이에 검증 로직을 의무화.

### 4-2. 세부 실행 계획
*   **파일 수정**: `validate_chunks_v2.py` 및 전체 통합 스크립트.
*   **CI/CD 및 젠킨스/크론 로직 흉내**:
    1.  현재 `validate_chunks_v2.py`의 출력 요약을 JSON으로 리턴받도록 인터페이스 수정.
    2.  Phase 2 로더(Step 6 적재 과정 등)가 종료된 가장 마지막 단계에서 `validate_chunks_v2.py`가 자율 실행되도록 `subprocess.call()` 삽입.
    3.  스크립트를 실행하여 반환된 `Total missing tables`가 100을 초과하거나(100인 기준 확립), 이전에 통과했던 주요 섹션(예: `8-3-8`)이 `FAIL`로 표시되면, 전체 파이프라인의 에러를 뿜으며 즉시 프로세스를 잠금. 
    4.  slack 알림 기능 또는 터미널 컬러 알림(Red Text) 기능 탑재 등을 고려.
